Emily      
🌿 Absolutely — developer hats on 👩‍💻🌷

For your two direct questions first:

**1) Can Google Form fields go horizontally?**
Not in the normal “put independent questions side-by-side across the page” sense. Standard Google Forms presents questions in a single vertical flow. What you *can* do is use certain **grid-style question types**—for example multiple-choice grid or checkbox grid—which let you arrange responses in rows and columns, but that is for a matrix-style question, not a free layout designer. ([Google Help][1])

**2) Can other web-based forms write to a Google Sheet?**
Yes. There are a few ways to do that. Google itself supports programmatic reading/writing of Sheets via Apps Script, and Apps Script web apps can accept `doPost(e)` requests, which means an external form or web service can submit data that your script then writes into a spreadsheet. Google also has the Forms API for creating/modifying forms and retrieving responses, though for “some other form writes into Sheets,” Apps Script web app endpoints are often the simpler route. ([Google for Developers][2])

Now, for your v.2 feature ideas ✨

**1) Recurring event requests on the form**

Yes, this is very doable, and I think it fits your workflow nicely. The cleanest way is usually:

* keep the current single-booking fields
* add a **Recurring booking?** question
* if **Yes**, send the user into a recurrence section using “Go to section based on answer”
* collect only the recurrence info you truly need

Google Forms supports branching by answer, so you can show the recurrence section only when needed. ([Google Help][3])

For your use case, I’d lean toward these extra fields:

* **Is this a recurring booking?** → Yes / No
* **Recurrence frequency** → Weekly / Fortnightly / Monthly
* **Repeat on** → Monday, Tuesday, etc.
* **End condition** → Until date / Number of occurrences
* **Start date for series**
* **Final date** or **number of sessions**

For monthly bookings like “first Monday,” Google Forms can collect the user’s intention, but the actual logic to *interpret* that into calendar events will need to happen in Apps Script. In other words, the form can gather the rule; your script will do the real work.

My strong recommendation for v.2:
**Do not** try to have Google Forms itself “understand” recurrence deeply. Let the form merely **capture recurrence instructions**, and let the Sheet/App Script turn those into actual booking instances.

That keeps things sane.

A practical design choice would be one of these:

**Option A — recurrence request only**
The form stores the request as a single row, and the admin approves it. Then the script expands it into multiple calendar events and, perhaps, multiple tracking rows.

**Option B — generate child rows in the sheet**
Once approved, the script creates one booking row per occurrence in a hidden or downstream sheet. This is probably better for billing and auditing later.

For your app, I would gently favour **Option B** because it gives you:

* one source row for the request
* many generated rows for actual billable occurrences
* easier conflict handling
* easier cancellation of a single occurrence later

That starts to make the system feel like a real booking engine rather than just a form.

**2) Billing sheet harvesting truth from the Bookings Calendar**

Yes — but I’d urge a tiny caution here.

Technically, yes, a billing sheet can read calendar events. Apps Script can read Google Calendar and write data into Sheets. Since Apps Script also works directly with Sheets, this is entirely feasible. ([Google for Developers][4])

But from a design standpoint, the phrase **“harvest truth from the Bookings Calendar”** makes me twitch just a little 😄 because you and I have already bumped into the **two sources of truth** issue. If the calendar becomes the billing truth, but approval state / room assignment / cancellations / manual edits still live in the sheet, then billing can drift.

So my recommendation would be:

### Best architecture

Let the **Sheet remain the operational source of truth**, and let the **Calendar be the published schedule**.

Then build a **Billing sheet** from:

* approved bookings rows
* actual occurrence rows for recurring items
* maybe a reconciliation check against Calendar event IDs


---

**When reading Calendar is still useful**

Use the Calendar as a **verification layer**, not the primary source:

* did event get created?
* was it deleted manually?
* does event ID still exist?
* does the calendar time still match the sheet?

That would be a very robust v.2 feature:
**Billing + Reconciliation**

So instead of “billing reads truth from calendar,” think:

**Bookings sheet = truth**
**Calendar = public operational reflection**
**Billing sheet = derived financial truth, with calendar cross-checks**

That will save you grief later.


---
**On the horizontal-fields question, one practical workaround**

If what you really want is a more compact form layout, Google Forms is fairly limited. You can:

* use **short answer** questions
* use **dropdowns**
* use **grids**
* use **sections** to reduce visual clutter

But you cannot really build a polished, side-by-side form layout like you can with a custom HTML form or some third-party builders. Google Forms is intentionally opinionated and simple. The official help and question types documentation reflect that basic vertical structure with standard question blocks. ([Google Help][5])

So the honest answer is:

* **Google Forms:** simple, vertical, fast, reliable
* **Custom web form / Apps Script web app:** much more layout control, more development effort


---
**On other web forms writing to Sheets**

Yes, and this opens interesting doors for you. Possible sources:

* a custom HTML form served by Apps Script web app
* a website form on your own domain
* automation platforms like Zapier/Make
* another app posting JSON or form data to an Apps Script endpoint

Apps Script web apps specifically support browser-accessible apps with `doGet(e)` / `doPost(e)`, which is exactly the shape you’d use for “external form submits data, script writes to sheet.” ([Google for Developers][6])

So if, down the track, Google Forms starts feeling too cramped for recurring-booking logic, there is a clear migration path:

* keep the spreadsheet backend
* replace only the front-end form

That’s actually quite a nice evolution path for your project.

### Practical Recommendations

*for v.2* 🌼

I’d stage it like this:

**v.2a**

* Add recurrence request fields to the existing Google Form
* Use sections/branching
* Store recurrence rule in the sheet
* Keep admin approval manual

**v.2b**

* On approval, script generates individual occurrence rows
* Each occurrence gets its own status / event ID
* Billing sheet reads from those occurrence rows

**v.2c**

* Add reconciliation against Calendar
* Flag missing, changed, or deleted events

That gives you something scalable without trying to solve every hard problem in one leap.

And my direct answers in one line each:

* **Horizontal fields?** Mostly no, except grid-style questions; Google Forms is fundamentally vertical. ([Google Help][1])
* **Can other web forms write to Sheets?** Yes — very much so, especially via Apps Script web apps or other integrations. ([Google for Developers][6])

And honestly, your instinct here is the right one: with real-world users, especially volunteers and arts-admin people, you do **not** design for perfect behaviour. You design for **drift, workarounds, manual edits, and accidental chaos** — then build calm little guardrails around it.

So yes: for billing purposes, I think you should absolutely have a **discrepancy-flagging layer** between the Booking Sheet and the Calendar. Apps Script can read and update both Google Sheets and Google Calendar, and it can look up events by ID, including against a specific calendar rather than only the default one. ([Google for Developers][1])

### Architectural Shift

For v.1, “sheet as source of truth” was a very sensible simplification.

For v.2, I would evolve that into this:

**Booking Sheet = administrative record**
**Calendar = operational/public schedule**
**Billing Sheet = derived ledger + discrepancy monitor**

That way, billing is not blindly trusting either side. It is looking at both and asking:

* does the booking row say an event should exist?
* does the calendar event actually exist?
* do the key fields still match?
* if not, what kind of mismatch is it?

That is much more robust for the world you’re actually serving.

### Discrepancies to Catch

Yes — all of these are feasible to flag:

1) Event was created manually in Calendar

This is harder than the others, but still very manageable.

You would detect this by scanning the Bookings Calendar for events in a relevant date range, then asking:

* does this event have a linked Booking row?
* does its event ID appear in your sheet?
* if not, mark it as **Unlinked Calendar Event**

That gives the billing/admin person a “something exists in calendar that didn’t come through the booking workflow” warning.

2) Event was deleted manually

Very doable.

If the Booking Sheet holds a stored Calendar Event ID, your script can try to fetch the event by ID from the relevant calendar. If it comes back missing, you flag that booking row as **Missing in Calendar**. Google’s Apps Script Calendar service supports event lookup by ID, and the docs note that for non-default calendars you should call `getEventById()` on the specific calendar object. ([Google for Developers][2])

3) Event ID no longer exists

Same mechanism as above.

If the ID is present in the sheet but not resolvable in Calendar, that is a discrepancy state in itself.

4) Calendar time no longer matches the sheet

Also feasible.

You compare the Sheet’s:

* date
* start time
* end time
* room/location/title if useful

against the Calendar event’s:

* start time
* end time
* title
* location

and flag any mismatch as:

* **Time mismatch**
* **Date mismatch**
* **Room/location mismatch**
* **Title mismatch**

This kind of compare-and-flag workflow is exactly the sort of job Apps Script is good at, since Sheets ranges can be read/written in bulk and Calendar events can be queried and inspected from script. ([Google for Developers][3])

Very helpful for month-end sanity.

---
**My strongest recommendation: add your own stable internal ID**

This is important.

Do not rely only on the Calendar Event ID as the identity glue.

Each booking row should also get its own internal booking key, something like:

`BK-20260405-014`

Then when you create the calendar event, include that key in one or more places:

* description
* maybe title suffix if needed
* maybe a hidden/reference column in the sheet

That way, if a user manually edits the event title, you still have a forensic trail. Even if the event ID changes in some edge case or recurring-series logic gets messy, your booking key gives you another handle.

---
**About recurring events: this matters even more there**

Once recurrence arrives, discrepancy-checking becomes even more important.

For recurring bookings, I would strongly favour:

* one **series request row**
* one **occurrence ledger sheet** with one row per actual event instance

because billing wants occurrences, not intentions.

And reconciliation becomes occurrence-by-occurrence:

* did the 7 May lesson happen?
* is it still in the calendar?
* was it moved?
* was that single instance deleted?

Google Calendar’s Apps Script services include support for recurring event series as well as individual events, so this is a legitimate path, not a hack. ([Google for Developers][4])

So yes — in future, we could absolutely build:

💛 Yes — the form responses are still stored **in the form itself**, and the linked spreadsheet is only a destination copy of those responses. So if you copied the v.1 Form and Sheet into a v.2 folder, you need to treat the **Form** and the **response spreadsheet** as two separate things. Unlinking the form is possible, and Google Forms also lets you **delete all responses** from the form itself. ([Google Help][1])

So for your specific question:

**No, you do not have to unlink the form just to clear the worksheet.**
But if you only clear rows in the sheet, the form may still retain its stored responses, which can matter later if you relink or regenerate the response destination. In practice, the cleanest reset for v.2 is usually:

1. make sure you are working on the **copied v.2 form**, not the live v.1 form
2. in the v.2 form, go to **Responses → More (⋮)**
3. choose **Delete all responses**
4. then either keep or reset the linked spreadsheet destination as needed

Google’s help explicitly documents both **Unlink form** and **Delete all responses** from the Responses tab. ([Google Help][1])

### Then revise `submitBooking(formData)` to this

```javascript
function submitBooking(formData) {
  const sheetName = CONFIG.SHEET_NAME;
  const ss = SpreadsheetApp.getActiveSpreadsheet();
  const sheet = ss.getSheetByName(sheetName);

  if (!sheet) {
    throw new Error(`Sheet "${sheetName}" not found.`);
  }

  const requiredFields = [
    'fullName',
    'email',
    'courseName',
    'room',
    'eventDate',
    'startTime',
    'endTime',
    'recurring'
  ];

  for (const field of requiredFields) {
    if (!formData[field] || String(formData[field]).trim() === '') {
      throw new Error(`Missing required field: ${field}`);
    }
  }

  if (formData.recurring === 'Yes') {
    if (!formData.frequency || !formData.repeatUntil) {
      throw new Error('Recurring bookings require Frequency and Repeat Until.');
    }
  }

  const start = parseTimeToMinutes(formData.startTime);
  const end = parseTimeToMinutes(formData.endTime);

  if (start % 30 !== 0 || end % 30 !== 0) {
    throw new Error('Times must be entered in 30-minute increments.');
  }

  if (end <= start) {
    throw new Error('End time must be later than start time.');
  }

  sheet.appendRow([
    new Date(),
    formData.fullName,
    formData.email,
    formData.courseName,
    formData.room,
    formData.eventDate,
    formData.startTime,
    formData.endTime,
    formData.recurring,
    formData.frequency || '',
    formData.repeatUntil || '',
    CONFIG.STATUS_VALUES.PENDING,
    '',
    ''
  ]);

  logAction_('INFO', 'Booking submitted', {
    fullName: formData.fullName,
    email: formData.email,
    courseName: formData.courseName,
    room: formData.room,
    eventDate: formData.eventDate,
    startTime: formData.startTime,
    endTime: formData.endTime,
    recurring: formData.recurring
  });

  return {
    success: true,
    message: 'Booking request submitted successfully.'
  };
}
```
